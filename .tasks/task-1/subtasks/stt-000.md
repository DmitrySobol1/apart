## Goal

Explore the Bnovo public API with real credentials, save raw responses as fixtures, create backend environment config, and document the actual data structure to validate/update plan assumptions.

## Details

- Curl all 5 Bnovo public API endpoints with real identifiers:
  - `GET /rooms?account_id=22720&dfrom=01-03-2026&dto=03-03-2026`
  - `GET /roomtypes?account_id=22720`
  - `GET /plans?account_id=22720`
  - `GET /amenities`
  - `GET /accounts?uid=d0ce239f-df14-4aa8-8ccf-83036c8cbb01`
- Save raw JSON responses as `fixtures/bnovo-rooms.json`, `fixtures/bnovo-roomtypes.json`, `fixtures/bnovo-plans.json`, `fixtures/bnovo-amenities.json`, `fixtures/bnovo-accounts.json`.
- Create `backend/.env` (gitignored) with `BNOVO_UID`, `BNOVO_ACCOUNT_ID`, `BNOVO_API_BASE`, `PORT`.
- Document actual field structure in `fixtures/README.md`:
  - Which endpoint contains photos, descriptions, amenities, prices, availability.
  - How room types map to rooms; how rate plans link to prices.
  - Whether `/rooms` already includes all data or if `/rooms` + `/roomtypes` merge is needed.
  - Where `totalPrice` comes from.
  - Whether amenities need a separate endpoint call.
- **Post-step:** Review and update stt-002 (backend proxy architecture) and stt-004 (frontend types and data mapping) if real data differs from plan assumptions.

- No code changes; manual exploration and documentation only.
