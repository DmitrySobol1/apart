## Issues Found

### MINOR

**MINOR-001:** `GuardedRoute` only guards `/rooms` and `/booking`, not `/confirmation`. Direct navigation to `/confirmation` without context will render the page with empty/null data (no room name, no dates shown, no guest shown). The page handles null gracefully (conditional rendering), so it does not crash, but it shows an incomplete success screen. The plan states "Navigating directly to `/rooms` or `/booking` without prior search redirects to `/`" — `/confirmation` is not explicitly listed as requiring a guard. Impact is cosmetic only.

**MINOR-002:** The plan specified `react-datepicker` as the date-picker library. The implementation uses native `<input type="date">`. The native control lacks the ability to show a minimum selectable date in the calendar UI on all browsers (the `min` attribute is set correctly, but users can still type invalid dates in some browsers). Functionally, the auto-correction logic in `DatePicker` component handles check-in ≥ check-out. This is an acceptable pragmatic choice that reduces bundle size and dependencies.

**MINOR-003:** `BookingSummary` mutates the `photos` array in-place with `.sort()` rather than `[...photos].sort()`. This is a minor React best-practice issue (avoid mutating props), although it has no observable runtime impact since the array is not used elsewhere after this component renders.

**MINOR-004:** The amenity-fetch error in `RoomsPage` is silently swallowed (`.catch(() => {})`). Amenity definitions not loading means room cards display without amenity icons/labels. A warning log would aid debugging in production.

**MINOR-005:** The `.env.example` is at `backend/.env.example` rather than the root `.env.example` specified in the plan's project structure section. The plan listed `├── .env.example` at the monorepo root. Minor structural deviation; all required variables are present in the file.

### КРИТИЧЕСКИЕ

### НЕЗНАЧИТЕЛЬНЫЕ

**MINOR-001:** `GuardedRoute` защищает только `/rooms` и `/booking`, но не `/confirmation`. Прямой переход на `/confirmation` без контекста отрисует страницу с пустыми/null данными (без имени номера, дат, гостя). Страница обрабатывает null корректно (условный рендеринг), поэтому не падает, но показывает неполный экран успеха. План указывает защиту только для `/rooms` и `/booking` — `/confirmation` явно не указан. Влияние только косметическое.

**MINOR-002:** План указывал `react-datepicker` как библиотеку выбора дат. Реализация использует нативный `<input type="date">`. Нативный контрол не может показать минимальную выбираемую дату в UI календаря во всех браузерах, но логика автокоррекции в компоненте `DatePicker` обрабатывает случай заезд ≥ выезд. Допустимый прагматичный выбор, уменьшающий размер бандла и зависимости.

**MINOR-003:** `BookingSummary` мутирует массив `photos` на месте через `.sort()` вместо `[...photos].sort()`. Незначительная проблема с точки зрения best practices React (избегать мутации props), хотя не имеет наблюдаемого влияния на runtime.

**MINOR-004:** Ошибка загрузки удобств в `RoomsPage` молча поглощается (`.catch(() => {})`). Если определения удобств не загрузились, карточки номеров отображаются без иконок/названий удобств. Предупреждение в логе помогло бы при отладке в продакшене.

**MINOR-005:** `.env.example` находится в `backend/.env.example`, а не в корневом `.env.example`, как указано в секции структуры проекта плана. Незначительное структурное отклонение; все необходимые переменные присутствуют в файле.
