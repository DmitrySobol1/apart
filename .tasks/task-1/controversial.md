## 1. `.memory-bank/` описывает ДРУГОЙ проект (критично)

**Проблема:** Вся папка `.memory-bank/` содержит документацию для проекта **"TG Channel Stats"** (Telegram-статистика каналов), а не для виджета бронирования апарт-отеля:

**Комментарий владельца:**
у нас на текущий момент есть схема для нашего проекта виджета бронирования? Есть есть - то замени информацию в data-model.md на нашу, если нет - то добавь в subtasks сделать это, когда у нас появится схема

**Отчёт по выполнению:**
Схемы для виджета бронирования в MVP нет — база данных не используется (данные логируются в консоль). Файл `data-model.md` переписан: старая SQLite-схема удалена, добавлен плейсхолдер с описанием планируемых MongoDB-коллекций (bookings, settings, logs) для post-MVP. Полная схема будет спроектирована при подключении MongoDB.

- `architecture/index.md` — ссылается на "TG Channel Stats project", SQLite + Drizzle ORM.
**Комментарий владельца:** поменяй данные на наши. В качестве БД будет mongoDB

Файл `architecture/index.md` полностью переписан под apart-nn: архитектурная диаграмма (Frontend → Backend → Bnovo API), Quick Reference с актуальным стеком (React 18, Express, Axios, dotenv), MongoDB + Mongoose указан как БД для post-MVP.

**Рекомендация:** Обновить `draft.md` — убрать MongoDB/Mongoose, или уточнить, нужна ли БД для MVP.

**Комментарий владельца:**
БД MongoDB/Mongoose будет использовать в проекте дальше при разработке, не в MVP версии

**Отчёт по выполнению:**
Принято. MongoDB/Mongoose остаётся в draft как часть общего стека проекта (post-MVP). В plan.md и steerings добавлены пометки: "Post-MVP: MongoDB + Mongoose". В `data-model.md` описаны планируемые коллекции. В MVP база данных не используется — это отражено во всех документах.

## 5. Конфиги агентов ссылаются на несуществующие команды

**Проблема:** Субагенты в `.claude/agents/` (Validator, Code Implementer и др.) скорее всего ссылаются на команды `pnpm qc`, `pnpm build`, `pnpm test:run` — унаследованные от TG Channel Stats. Эти команды не существуют в проекте apart-nn и будут падать, пока не будут настроены скрипты в `package.json` при выполнении stt-001.

**Рекомендация:** После stt-001 (инициализация проекта) обновить конфиги агентов под реальные скрипты из `package.json`. Либо прописать ожидаемые скрипты в критериях приёмки stt-001, чтобы агенты заработали сразу.

**Комментарий владельца:**
После stt-001 (инициализация проекта) обновить конфиги агентов под реальные скрипты из `package.json`. Добавь это в задачи, чтобы у нас в проекте были аналогичные команды для агентов. Они же для тестов сделаны?

**Отчёт по выполнению:**
Нет, агенты — это не только для тестов. Вот полный список и их роли:
- **Code Implementer** — основной исполнитель: пишет код по подзадачам, запускает `qc` после реализации
- **Code Writer** — пишет код по прямым инструкциям (вне workflow подзадач), запускает `qc`
- **Validator** — запускает `qc` + `build`, проверяет что код собирается и проходит линтинг
- **Test Writer** — только пишет тесты, НЕ запускает их
- **Auditor** — проверяет код по плану, запускает `test:run` для определения TEST_STATUS
- **Documentation Writer** — пишет/обновляет .md документацию

Все агенты, кроме Test Writer и Docs Writer, зависят от команд из `project-commands.md`. Обновление agent configs + hooks + project-commands включено в подзадачу **stt-010**. Агенты читают команды из `project-commands.md`, поэтому после обновления этого файла в stt-010 все агенты заработают корректно без изменений в их собственных промптах (кроме пути `apps/specs/index.md` в code-implementer, который тоже будет исправлен в stt-010).

## 6. Hooks могут не работать

**Проблема:** `.claude/hooks.json` скорее всего настроен на команды из проекта TG Channel Stats. Пока apart-nn не инициализирован и не имеет соответствующих скриптов, hooks будут падать при каждом вызове инструментов.

**Рекомендация:** Либо отключить hooks до завершения stt-001, либо обновить их заранее на no-op / условную проверку существования проекта.

**Комментарий владельца:**
После stt-001 (инициализация проекта) обновить hooks, чтобы они работали аналогично проекту TG Channel Stats

**Отчёт по выполнению:**
Hooks (`hooks.json`) оставлены как есть на данный момент. Они будут падать до завершения stt-001, но это не блокирует работу — hooks не являются обязательными. Обновление hooks включено в критерии приёмки подзадачи **stt-010**:
- `pnpm qc:wrapped` → будет заменён на актуальную команду QC
- `pnpm code-style` → будет заменён на команду форматирования (e.g., `npx prettier --write`)
- Функциональность stop-hook.sh сохранится (защита от infinite loops, автоисправление ошибок)
