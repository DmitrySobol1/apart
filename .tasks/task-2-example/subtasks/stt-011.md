# fix / Web dashboard — metrics timestamp alignment + default 30d range — consistent API contracts

## Goal

Align the web dashboard with backend API contracts by (1) switching metrics time key to `tsHour` across types and chart bindings, and (2) enforcing a default last-30-days date range when `from`/`to` are absent on posts list and post detail. Also harmonize env var naming/docs for the web dev/prod proxy.

## Tech Requirements

- Target: `packages/web`
- Stack: React, TypeScript, Vite, SWR, Recharts, Tailwind
- Data model: Conform to backend schema (`PostMetricsPoint.tsHour`) — do not introduce ad-hoc fields
- Where to change:
  - Types: `src/types/posts.ts` — rename `timestamp` → `tsHour` in `PostMetricsPoint`
  - Chart: `src/components/MetricsChart.tsx` — XAxis `dataKey` → `tsHour`, tooltip label formatting uses `tsHour`
  - Metrics hook: `src/hooks/usePostMetrics.ts` — no shape remap; ensure types flow remains consistent
  - Default range behavior:
    - Posts list: If `from`/`to` are missing in URL, auto-set to last 30 days and clear `offset` without user action
      - Prefer centralized handling: either in `DateRangePicker` (on mount) or in `usePosts` before building the key
    - Post detail: If `from`/`to` missing, fetch metrics for last 30 days by default (via state or URL sync)
  - Config/docs: Align env var naming in docs to `VITE_API_URL` (current code) or switch code to `VITE_API_BASE_URL` consistently; update `.env.example` and README accordingly
- Dependencies: None on DB/backend modifications; uses existing endpoints
- Constraints: ≤600 LOC per file; ≤3–4 hours SWE effort
- Conventions: Follow `.memory-bank/steering/development-conventions.md` (strict types, reuse schemas), and `.memory-bank/steering/decompose-conventions.md`

## Acceptance Criteria

- Metrics timestamp alignment
  - `packages/web/src/types/posts.ts` defines `PostMetricsPoint` with `tsHour: ISODateString`
  - `MetricsChart` X-axis binds to `tsHour`; tooltips show correct local date/time
  - App renders the post detail chart without runtime errors; series render as expected

- Default last-30-days behavior
  - `/channels/:username` loads posts for the last 30 days when URL lacks `from`/`to` (no manual Apply required)
  - `/channels/:username/posts/:messageId` loads metrics for the last 30 days when URL lacks `from`/`to`
  - Applying filters resets pagination `offset` to 0 and preserves `from`/`to` in URL

- Config/docs consistency
  - `.env.example` and README match the env var used by code (`VITE_API_URL` or `VITE_API_BASE_URL`, pick one and standardize)
  - Vite dev proxy and prod `server.js` proxy instructions documented briefly

- Quality checks
  - `pnpm --filter @tg-channel-stats/web build` and `type-check` pass
  - Manual validation: channels → posts list default range; posts list pagination dedupe; post detail chart time axis shows correct local timestamps
